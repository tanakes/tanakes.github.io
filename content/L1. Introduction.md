# 0. Data Models & The Relational Model

A **data model** is a collection of tools to describe data, relationships, semantics, and constraints.

- **Common Models:** Relational (dominant), Entity-Relationship (for design), Object-based, Semi-structured (XML), and older Hierarchical/Network models.
- **Relational Model:** Stores all data in **tables** (relations) made of rows and columns.
    - _Example:_ An `instructor` table with columns `ID`, `name`, `dept_name`, `salary`. Each row represents one instructor.

# 1. Instance 🧩

An **instance** is the **actual data stored in the database at a specific moment in time**.

Think:

- Schema = structure
    
- Instance = current contents
    

---

## Example

Schema:

```sql
CREATE TABLE students (
    id INT,
    name TEXT
);
```

Current data:

|id|name|
|---|---|
|1|Ali|
|2|Sara|

This current data is the **instance**.

If data changes tomorrow, the instance changes too.

---

# 2. Physical Schema 💾

The **physical schema** describes:

- how data is physically stored on disk,
    
- file organization,
    
- indexing methods,
    
- compression,
    
- storage structures.
    

Low-level details.

---

## Examples

- B-trees
    
- Hash indexes
    
- File blocks
    
- Disk pages
    

Users usually do not see this.

---

# 3. Logical Schema 🧠

The **logical schema** describes:

- tables,
    
- columns,
    
- relationships,
    
- constraints.
    

This is the main database design.

---

## Example

```sql
CREATE TABLE employees (
    id INT PRIMARY KEY,
    name TEXT,
    salary NUMERIC
);
```

This is logical structure.

---

# 4. Views 👁️

A **view** is a **virtual table** created from a query.

It does not usually store data itself.

---

## Example

```sql
CREATE VIEW high_salary AS
SELECT name, salary
FROM employees
WHERE salary > 5000;
```

Now:

```sql
SELECT * FROM high_salary;
```

works like a table.

---

# Why views are useful

- Security 🔒
    
- Simpler queries
    
- Hide complexity
    
- Customized access
    

---

# 5. DDL (Data Definition Language)

DDL defines database structure.

---

## Common DDL commands

|Command|Purpose|
|---|---|
|`CREATE`|create objects|
|`ALTER`|modify objects|
|`DROP`|delete objects|
|`TRUNCATE`|remove all rows|

---

## Example

```sql
CREATE TABLE students (
    id INT,
    name TEXT
);
```

---

# 6. DML (Data Manipulation Language)

DML manipulates data.

---

## Common DML commands

|Command|Purpose|
|---|---|
|`SELECT`|read|
|`INSERT`|add|
|`UPDATE`|modify|
|`DELETE`|remove|

---

## Example

```sql
INSERT INTO students
VALUES (1, 'Ali');
```

---

# 7. Procedural vs Declarative ⚖️

---

## Procedural Language

You specify:

- **what** to do,
    
- and **how** to do it.
    

Like step-by-step instructions.

---

### Example

Traditional programming languages:

- C
    
- Java
    

---

## Declarative Language

You specify:

- **what result you want**,
    
- not the exact process.
    

SQL is mostly declarative.

---

### Example

```sql
SELECT * FROM employees
WHERE salary > 5000;
```

You do not tell PostgreSQL:

- how to scan disk,
    
- which algorithm to use,
    
- how indexes work.
    

The DBMS decides.

---

# 8. Database Engine ⚙️

The **database engine** is the core software that:

- stores data,
    
- processes queries,
    
- manages transactions,
    
- controls concurrency,
    
- retrieves records.
    

It is basically the "brain" of the DBMS.

---

# 9. Storage Manager 💽

The **storage manager** handles physical storage operations.

It sits between:

- disk storage,
    
- and higher DBMS components.
    

---

## Components

---

## A) Data Files

Actual files storing table data.

Contains rows/records.

---

## B) Data Dictionary 📚

Metadata storage.

Stores information about:

- tables,
    
- columns,
    
- constraints,
    
- users,
    
- permissions,
    
- indexes.
    

Also called:

- system catalog.
    

---

### Example metadata

```text
Table: employees
Columns: id, name, salary
Primary Key: id
```

---

## C) Indices (Indexes)

Special structures that speed up searching.

Without index:

```text
Search every row
```

With index:

```text
Jump directly to data
```

---

### Example

```sql
CREATE INDEX idx_salary
ON employees(salary);
```

---

# 10. Query Processor 🧠

The query processor handles SQL queries.

It converts SQL into efficient execution steps.

---

# Components of Query Processor

---

# A) DDL Interpreter

Processes DDL commands.

Example:

```sql
CREATE TABLE students (...);
```

It:

- updates metadata,
    
- creates structures.
    

---

# B) DML Compiler

Processes DML queries.

Example:

```sql
SELECT * FROM students;
```

Converts SQL into low-level instructions.

---

# C) Query Evaluation Engine

Actually executes the query plan.

Reads data, applies filters, joins tables, etc.

---

# Query Processing Stages

---

## 1. Parsing and Translation

Checks:

- syntax,
    
- semantics,
    
- table existence,
    
- column names.
    

Converts SQL into internal representation.

---

## Example

```sql
SELECT name FROM employees;
```

DBMS verifies:

- table exists,
    
- column exists.
    

---

## 2. Optimization 🚀

DBMS searches for the fastest execution plan.

Chooses:

- indexes,
    
- join methods,
    
- scan methods.
    

---

## Example

Should it:

- scan full table?
    
- use index?
    
- hash join?
    
- merge join?
    

Optimizer decides.

---

## 3. Evaluation

Actually runs the optimized plan.

Returns results.

---

# 11. Transaction Management 🔒

A **transaction** is a group of operations treated as one unit.

Either:

- all succeed,
    
- or all fail.
    

---

## Example

Bank transfer:

```text
Subtract from A
Add to B
```

Cannot allow:

- money removed from A,
    
- but not added to B.
    

---

# ACID Properties

|Property|Meaning|
|---|---|
|Atomicity|all or nothing|
|Consistency|valid state maintained|
|Isolation|concurrent transactions safe|
|Durability|committed data survives crash|

# 12. Two-Tier Architecture 🏗️

Client communicates directly with database server.

---

## Structure

```text
Client Application ↔ Database Server
```

---

## Example

Desktop app directly connecting to PostgreSQL.

---

## Pros

- Simple
    
- Fast
    

## Cons

- Less scalable
    
- Security issues
    

---

# 13. Three-Tier Architecture 🏢

Adds middle layer.

---

## Structure

```text
Client
   ↕
Application Server
   ↕
Database Server
```

---

# Layers

|Layer|Purpose|
|---|---|
|Presentation|UI|
|Application|business logic|
|Database|storage|

---

# Example

Browser → Backend API → PostgreSQL

---

# Advantages

- Better security 🔒
    
- Easier maintenance
    
- Scalable
    
- Centralized logic
    

---

# Big Picture 🌐

```text
Users
  ↓
Applications
  ↓
Query Processor
  ↓
Storage Manager
  ↓
Disk Storage
```
