### 🔹 1. Basic Concepts of Indexing
Indexes are data structures that speed up data retrieval, much like a library's card catalog.
- **Search Key:** The attribute (or set of attributes) used to look up records.
- **Index File:** Stores `(search-key, pointer)` pairs. It's significantly smaller than the main data file, allowing the DB to find records without scanning every row.
- **Two Fundamental Types:**

| Type                | How it works                                          | Best For                                |
| ------------------- | ----------------------------------------------------- | --------------------------------------- |
| **Ordered Indices** | Search keys stored in sorted order                    | Range queries, sorting, mixed workloads |
| **Hash Indices**    | Hash function distributes keys uniformly into buckets | Exact-match lookups ( = )               |
- **Evaluation Metrics:** Access time, insertion/deletion time, supported query types, and space overhead.

---

### 🔹 2. Ordered Indices: Clustering, Density & Structure
| Concept | Definition | Example / Rule |
|---------|------------|----------------|
| **Clustering (Primary) Index** | Search key matches the physical/sequential order of the file on disk | `instructor` table physically sorted by `ID` → index on `ID` is clustering |
| **Secondary (Nonclustering) Index** | Search key differs from physical file order | Index on `salary` when file is sorted by `ID` |
| **Dense Index** | Contains an entry for **every** search-key value in the file | Required for secondary indices; pointer points to a bucket of matching records |
| **Sparse Index** | Contains entries for **only some** search-key values | Only valid if file is physically ordered on the key. To find `K`, locate largest index entry `< K`, then scan sequentially |

#### 📚 Multilevel Indices
If an index grows too large for memory, the DB builds a **sparse index on the index itself** (outer index). This creates a hierarchy of indices, keeping the top level in memory for fast traversal.

---

### 🔹 3. B+-Tree Index Files (The Industry Standard)
B+-trees are balanced, multi-level sparse indices optimized for disk I/O.

#### 🌲 Core Properties
- **Balanced:** All paths from root to leaf are the same length.
- **Node Capacity:** Non-leaf nodes (except root) have `⌈n/2⌉` to `n` children. Leaf nodes have `⌈(n-1)/2⌉` to `n-1` values.
- **Leaf Linking:** Leaves are chained in sorted order → enables fast **range queries**.
- **Height:** `⌈log⌈n/2⌉(K)⌉`. With 1M keys & `n=100`, height ≤ 4. (Binary trees would need ~20 levels, causing far more disk I/O).

#### ⚙️ B+-Tree Updates
| Operation | Process |
|-----------|---------|
| **Insertion** | 1. Find target leaf.<br>2. If room, insert.<br>3. If full, **split** node into two. Push middle key up to parent.<br>4. Repeat upward. Root split increases tree height by 1. |
| **Deletion** | 1. Remove entry from leaf.<br>2. If underfull, try to **borrow** from a sibling.<br>3. If can't borrow, **merge** with sibling and update parent.<br>4. If root ends up with 1 child, delete root & promote child. |

> 💡 **Why B+-Trees dominate:** Splits/merges are rare. Most inserts/deletes only touch a leaf node. Internal nodes stay cached in memory, making lookups extremely fast in practice.

---

### 🔹 4. Hashing
Hash indices map search keys directly to bucket addresses using a hash function `h(K)`.

#### 📦 Static Hashing
- Fixed number of buckets. Fast for exact matches, but suffers from:
  - **Bucket Overflow:** Caused by too few buckets or hash skew. Handled via **overflow chaining** (linked list of extra blocks).
  - **Size Mismatch:** If DB grows → heavy overflow & slow lookups. If DB shrinks → wasted space.
- Periodic reorganization is expensive and disruptive.

#### 🔄 Dynamic Hashing
Adjusts bucket count automatically as data changes:

| Technique              | How it Works                                                                                                                                |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| **Extendable Hashing** | Uses a directory of pointers. Splits buckets by consuming more bits of the hash value. Grows smoothly without doubling all buckets at once. |
| **Linear Hashing**     | Performs incremental rehashing, spreading the cost of reorganization across many insertions.                                                |
#### ⚖️ Ordered Indexing vs. Hashing
| Feature | B+-Tree (Ordered) | Hash Index |
|---------|-------------------|------------|
| **Exact Match (`=`)** | Fast (`O(log n)`) | Faster (`O(1)`) |
| **Range Queries (`>`, `BETWEEN`)** | Excellent (leaf chaining) | Poor (requires full scan) |
| **Sorting / ORDER BY** | Native support | Not supported |
| **Real-World Usage** | Default in Oracle, SQL Server, PostgreSQL | PostgreSQL discourages; rarely used as primary index |

### 🔹 5. Indices on Multiple Keys & Bitmap Indices
#### 🔗 Composite Search Keys
- Stored in **lexicographic order**: `(a1, a2) < (b1, b2)` if `a1 < b1` or `(a1 = b1 and a2 < b2)`.
- **Efficient for:** `WHERE dept_name = 'Finance' AND salary = 80000` or `dept_name = 'Finance' AND salary < 80000`.
- **Inefficient for:** `WHERE dept_name < 'Finance' AND salary = 80000` (fetches many irrelevant rows).

#### 🖼️ Bitmap Indices
Designed for attributes with **low cardinality** (few distinct values): e.g., `gender`, `status`, `state`, income brackets.
- **Structure:** One bitmap per distinct value. Each bit corresponds to a record (`1` = matches, `0` = doesn't match).
- **Query Execution:** Uses fast bitwise operations:
  - `AND` → Intersection
  - `OR` → Union
  - `NOT` → Negation
- *Example:* Find males with income level `L1`:
  `bitmap_male AND bitmap_L1 = 10000` → retrieve record 0.
- **Space Efficiency:** Extremely compact. For 8 distinct values, index is ~1% of the base table size. Ideal for data warehousing & OLAP.

---

### 💡 Key Takeaways
1. **Indexes trade write speed for read speed:** They accelerate `SELECT` queries but add overhead to `INSERT`/`UPDATE`/`DELETE`.
2. **B+-trees are the universal default:** They handle exact matches, ranges, sorting, and scale gracefully with disk I/O in mind.
3. **Hashing is specialized:** Excellent for primary-key lookups in memory/temp tables, but fails for ranges and struggles with dynamic sizing without extendable/linear variants.
4. **Composite keys follow left-to-right rules:** An index on `(A, B)` helps queries on `A` or `(A, B)`, but not standalone `B`.
5. **Bitmap indices power analytics:** Their bitwise operations make them blazingly fast for multi-condition filtering on categorical data in data warehouses.
6. **Real-world DBs optimize automatically:** Modern optimizers choose between index types, combine multiple indexes, or fall back to sequential scans based on selectivity and statistics.

> 🎯 **Next Step:** Understanding indexing directly enables **query optimization (Chapter 15)**. The optimizer uses index metadata (height, selectivity, clustering factor) to estimate costs and choose the fastest execution plan. Mastering B+-trees and composite keys is essential for writing index-aware SQL and avoiding performance bottlenecks in production systems.
