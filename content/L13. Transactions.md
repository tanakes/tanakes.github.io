---

### 🔹 1. Transaction Concept & ACID Properties
A **transaction** is a single logical unit of work that accesses and possibly updates multiple data items. It must guarantee database integrity even during system crashes or concurrent access.

#### 🛡️ The ACID Guarantee
| Property | Definition | Example / Why It Matters |
|----------|------------|--------------------------|
| **Atomicity** | All operations in a transaction execute completely, or none do. | Transferring $50 from Account A to B: If the system crashes after deducting from A but before adding to B, money is "lost". Atomicity ensures partial updates are rolled back. |
| **Consistency** | Transactions preserve all integrity constraints. The DB must be consistent before and after execution. | `Sum(A, B)` must remain constant after a transfer. Temporary inconsistency is allowed *during* execution, but not at commit. |
| **Isolation** | Concurrent transactions execute independently. Intermediate states are hidden from others. | If T2 reads `A+B` while T1 is midway through the transfer, T2 shouldn't see the temporarily reduced sum. |
| **Durability** | Once committed, changes persist permanently, even after hardware/software failures. | After a successful `COMMIT`, the $50 transfer survives a power outage. |

---

### 🔹 2. Transaction States & Concurrency
#### 🔄 State Lifecycle
`Active` → `Partially Committed` (final statement executed) → `Committed` (changes permanent)  
**OR** `Failed` (cannot proceed) → `Aborted` (rolled back to pre-transaction state) → `Restart` or `Kill`

#### ⚡ Why Run Transactions Concurrently?
- **Higher throughput:** CPU processes one transaction while another waits for disk I/O.
- **Lower latency:** Short transactions don't queue behind long ones.
- **Trade-off:** Requires strict **concurrency control** to prevent anomalies.

---

### 🔹 3. Schedules & Serializability
A **schedule** is a chronological sequence of read/write operations from one or more transactions.

| Schedule Type | Description | Outcome |
|---------------|-------------|---------|
| **Serial** | Transactions run one after another (`T1` then `T2`, or `T2` then `T1`) | Always consistent, but poor performance |
| **Serializable** | Concurrent schedule that produces the **same final state** as some serial schedule | Correct & high performance |
| **Non-Serializable** | Concurrent schedule that breaks consistency | ❌ `A + B` sum changes unexpectedly; data corruption |

#### 🔍 Conflict Serializability (The Practical Standard)
Two instructions **conflict** if they access the same data item `Q` and at least one is a `write`:
- `read(Q)` & `read(Q)` → ✅ No conflict
- `read(Q)` & `write(Q)` → ❌ Conflict
- `write(Q)` & `write(Q)` → ❌ Conflict

A schedule is **conflict serializable** if it can be transformed into a serial schedule by swapping **non-conflicting** operations.

> 💡 **View Serializability:** A weaker, theoretical form based on matching initial reads, intermediate writes, and final writers. *Every conflict-serializable schedule is view-serializable, but not vice versa.* Testing for view serializability is **NP-complete**, so DBMSs rely on conflict serializability.

---

### 🔹 4. Testing for Serializability: Precedence Graph
The DBMS uses a directed graph to verify correctness:
- **Vertices:** Transactions (`T1, T2, ...`)
- **Edges:** `Ti → Tj` if `Ti` conflicts with `Tj` and executes first on the same data item.

✅ **Rule:** A schedule is conflict serializable **if and only if** the precedence graph is **acyclic**.  
🔁 **Serializability Order:** Found via **topological sort** of the graph.  
*Example:* `T5 → T1 → T3 → T2 → T4` means executing them in this order yields the same result as the concurrent schedule.

---

### 🔹 5. Recoverability & Cascading Rollbacks
Concurrency control must also handle transaction failures gracefully.

| Concept | Rule | Purpose |
|---------|------|---------|
| **Recoverable Schedule** | If `Tj` reads data written by `Ti`, `Ti` must **commit** before `Tj` commits. | Prevents `Tj` from permanently using inconsistent data if `Ti` aborts. |
| **Cascading Rollback** | `Ti` aborts → forces rollback of all `Tj` that read `Ti`'s uncommitted data. | ❌ Wastes significant work; undesirable. |
| **Cascadeless Schedule** | `Tj` can only read data written by `Ti` **after** `Ti` commits. | ✅ Eliminates cascading rollbacks. Strongly preferred in practice. |

---

### 🔹 6. SQL Transaction Control & Isolation Levels
#### 📝 Transaction Boundaries in SQL
```sql
-- Begins implicitly with first statement
UPDATE accounts SET balance = balance - 50 WHERE id = 'A';
UPDATE accounts SET balance = balance + 50 WHERE id = 'B';
COMMIT WORK;  -- Makes changes permanent
-- OR ROLLBACK WORK; -- Undoes all changes in current transaction
```
- By default, most systems auto-commit after each statement. Disable via `connection.setAutoCommit(false)` (JDBC).

#### 🔒 SQL-92 Isolation Levels
| Level | Guarantee | Risk / Phenomenon |
|-------|-----------|-------------------|
| **Serializable** | Full ACID isolation; equivalent to serial execution | Highest safety, lowest concurrency |
| **Repeatable Read** | Repeated reads return same value; only committed data visible | May see "phantom" rows inserted by others |
| **Read Committed** | Only reads committed data | Repeated reads may return different values (non-repeatable reads) |
| **Read Uncommitted** | Can read uncommitted data | **Dirty reads** allowed; fastest but least safe |

> ⚠️ **Note:** Oracle & older PostgreSQL used **Snapshot Isolation** by default (not strictly in SQL standard but provides strong practical guarantees).

---

### 🔹 7. Implementation Mechanisms & The Phantom Problem
#### ⚙️ How DBMSs Enforce Isolation
1. **Locking:** Shared locks (readers), Exclusive locks (writers). Held for varying durations.
2. **Timestamps:** Assign start time to each transaction; reject out-of-order reads/writes.
3. **Multi-Version Concurrency Control (MVCC):** Keep historical versions of rows so readers get a consistent "snapshot" without blocking writers.

#### 👻 The Phantom Phenomenon
Occurs with **range/predicate queries** when using row-level locking:
```sql
-- T1: Lock all instructors with salary > 90000
SELECT * FROM instructor WHERE salary > 90000;

-- T2 executes concurrently:
INSERT INTO instructor VALUES('99999', 'Smith', 'Marketing', 95000);
```
- T1 and T2 **logically conflict** (T2 adds a row that matches T1's predicate), but T1 only locked existing tuples. T2 slips through → **Phantom**.
- ✅ **Solution:** **Predicate Locking** or **Index-Range Locking** (locking the gap/range, not just existing rows).

---

### 💡 Key Takeaways
1. **ACID properties** are non-negotiable for reliable database systems. Atomicity and Durability handle crashes; Consistency and Isolation handle logic and concurrency.
2. **Serializability** is the correctness criterion for concurrent execution. **Conflict serializability** is efficiently testable via precedence graphs and is the industry standard.
3. **Recoverable & cascadeless schedules** prevent aborted transactions from corrupting or invalidating the work of others.
4. **SQL isolation levels** offer a performance vs. safety tradeoff. `Serializable` is safest; `Read Committed` is common default for OLTP.
5. **The Phantom Problem** proves that row-level locks aren't enough for predicate-based queries; range/index locks or MVCC snapshots are required.
