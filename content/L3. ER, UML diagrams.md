| Type | Meaning | Notation in Slides |
|------|---------|-------------------|
| **1:1** | One entity links to at most one other | `A â†’ B` and `B â†’ A` |

### đź”ą 6. Weak Entity Sets
- **Definition:** Entities that **cannot be uniquely identified** by their own attributes alone. They depend on an identifying (strong) entity.
- **Representation:** 
  - Double rectangle (entity)
  - Dashed underline for the **discriminator** (partial key)
  - Double diamond for the **identifying relationship**
- **Primary Key:** `Identifying Entity's PK + Discriminator`
- *Example:* `section` depends on `course`. A section is uniquely identified by `(course_id, sec_id, semester, year)`. `course_id` is borrowed from the `course` entity.

> đź’ˇ Weak entities are "existence-dependent": if the parent entity is deleted, the weak entity must also be deleted.

### đź”ą 7. Converting E-R Diagrams to Relational Schemas (Tables)
| E-R Construct | Relational Translation | Example |
|---------------|------------------------|---------|
| **Strong Entity** | Table with same attributes | `instructor(ID, name, salary)` |
| **Weak Entity** | Table + identifying entity's PK | `section(course_id, sec_id, semester, year)` |
| **Composite Attr.** | Flattened into separate columns | `first_name, last_name` |
| **Multivalued Attr.** | New table: `(Entity_PK, Attribute)` | `inst_phone(ID, phone_number)` |
| **N:M Relationship** | Separate table with both PKs + attributes | `advisor(s_id, i_id, date)` |
| **1:N / N:1 (Total)** | Merge into "Many" side as FK (no new table) | Add `dept_name` FK to `instructor` |
| **1:1** | Merge into either table as FK | Add `office_id` to `instructor` |

> âš ď¸Ź Merging 1:N/1:1 relationships avoids extra tables but may introduce `NULL`s if participation is partial.

### đź”ą 8. Extended E-R Features
#### đźŚż Specialization & Generalization
- **Specialization (Top-down):** Sub-classing a higher-level entity.  
  *Example:* `Person` â†’ `Student`, `Instructor`. Lower-level inherits all attributes/relationships.
- **Generalization (Bottom-up):** Grouping similar entities into a super-entity.
- **Constraints:** 
  - *Disjoint:* An entity can belong to only one subclass.
  - *Overlapping:* Can belong to multiple.
  - *Total/Partial Completeness:* Must an entity belong to a subclass? (Default = partial)

#### đź”„ Aggregation
- **Problem:** Some relationships need to relate to *other relationships*, not just entities.
- **Solution:** Treat a relationship set as an **abstract entity** so new relationships can attach to it.
- *Example:* `proj_guide` links `student`, `instructor`, and `project`. We want to record an `evaluation` for that specific combination. Aggregation lets `evaluation` relate to the whole `proj_guide` relationship without duplicating data.

### đź”ą 9. Design Guidelines & Common Mistakes
| Guideline | Explanation |
|-----------|-------------|
| **Entity vs. Attribute** | Use an entity if the "attribute" has its own properties or relationships. *Example:* `phone` as entity if tracking type, location, or multiple numbers. |
| **Entity vs. Relationship** | Use relationships for associations/actions between entities. |
| **Binary vs. N-ary** | Prefer binary relationships unless a single action naturally requires 3+ entities simultaneously (e.g., `proj_guide`). N-ary can be split into binary + artificial entity, but may lose implicit constraints. |
| **Placement of Attributes** | Attach relationship attributes to the relationship set, not to one participating entity. *Example:* `date` of advising belongs to `advisor`, not `student`. |

#### đź†š E-R vs. UML Class Diagrams
- Both model entities, relationships, and inheritance.
- **Key Differences:** UML reverses cardinality notation placement, uses different generalization arrows, and includes software engineering constructs (methods, visibility). Database designers often translate UML to E-R before generating schemas.
