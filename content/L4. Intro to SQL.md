### 🔹 1. SQL Overview & Architecture
- **History & Standards:** Developed at IBM (originally SEQUEL → SQL). Standardized by ANSI/ISO (SQL-86 through SQL:2003). Commercial systems implement SQL-92+ with proprietary extensions.
- **SQL Components:**
  - `DML`: Query & modify data (`SELECT`, `INSERT`, `UPDATE`, `DELETE`)
  - `DDL`: Define schemas, constraints, views, indices, security
  - `Transaction Control`: Begin/commit/rollback transactions
  - `Embedded SQL & APIs`: JDBC/ODBC let host languages (Java, Python, C++) send SQL to the DB
- **Key Trait:** SQL is **declarative** (you specify *what* you want, not *how* to get it). It's intentionally not Turing-complete, so complex logic/UI/networking is handled in host applications.

### 🔹 2. Data Definition Language (DDL)
#### 📦 Domain Types
| Type | Description | Example |
|------|-------------|---------|
| `char(n)` | Fixed-length string | `'A'` padded to `n` chars |
| `varchar(n)` | Variable-length string (max `n`) | `'Smith'` |
| `int` / `smallint` | Machine-dependent integers | `100`, `-5` |
| `numeric(p,d)` | Fixed-point, `p` digits total, `d` decimal | `numeric(5,2)` → `123.45` |
| `real` / `float(n)` | Floating-point | `3.14159` |

#### 🏗️ Creating Tables & Constraints
```sql
CREATE TABLE instructor(
  ID char(5), 
  name varchar(20) NOT NULL, 
  dept_name varchar(20), 
  salary numeric(8,2), 
  PRIMARY KEY(ID), 
  FOREIGN KEY(dept_name) REFERENCES department
);
```
- **Integrity Constraints Enforced Automatically:** DB blocks any `INSERT`/`UPDATE`/`DELETE` that violates `PRIMARY KEY`, `FOREIGN KEY`, or `NOT NULL`.

#### 🛠️ Schema Updates
| Command | Purpose | Example |
|---------|---------|---------|
| `INSERT INTO r VALUES(...)` | Add a row | `INSERT INTO instructor VALUES('10211','Smith','Biology',66000);` |
| `DELETE FROM r` | Remove rows | `DELETE FROM student WHERE tot_cred < 0;` |
| `DROP TABLE r` | Delete table & data | `DROP TABLE temp_scores;` |
| `ALTER TABLE r ADD A D` | Add column | Existing rows get `NULL` for `A` |
| `ALTER TABLE r DROP A` | Remove column | Not universally supported |

### 🔹 3. Basic Query Structure: `SELECT-FROM-WHERE`
Maps directly to Relational Algebra: `SELECT` ≈ Projection, `WHERE` ≈ Selection, `FROM` ≈ Cartesian Product + Join.

#### 🔍 The `SELECT` Clause
- Lists desired columns. **Default allows duplicates**; use `DISTINCT` to remove them.
- Supports arithmetic & aliases:
  ```sql
  SELECT ID, name, salary/12 AS monthly_salary FROM instructor;
  ```
- `SELECT *` returns all columns. Literals work too: `SELECT 'A' FROM instructor` returns a column of `'A'` repeated for each row.

#### 📂 The `FROM` Clause
- Lists tables. Without a `WHERE` clause, it produces a **Cartesian product**.
- Common column names are prefixed: `instructor.ID`, `teaches.ID`

#### ⚙️ The `WHERE` Clause
- Filters rows using comparison operators (`=, <>, <, >, <=, >=`) and logical connectives (`AND, OR, NOT`).
- **Range & Tuple Comparison:**
  ```sql
  WHERE salary BETWEEN 90000 AND 100000
  WHERE (instructor.ID, dept_name) = (teaches.ID, 'Biology')
  ```

#### 🔄 The Rename Operation (`AS`)
- Essential for **self-joins** or simplifying complex queries. `AS` is optional.
  ```sql
  SELECT DISTINCT T.name FROM instructor AS T, instructor AS S 
  WHERE T.salary > S.salary AND S.dept_name = 'Comp. Sci.';
  ```

### 🔹 4. Advanced Filtering & Set Operations
#### 🔤 String Matching (`LIKE`)
- `%` = any substring, `_` = exactly one character. Case-sensitive.
- Escape special chars: `LIKE '100\%' ESCAPE '\'`
- Examples: `'Intro%'` (starts with), `'%Comp%'` (contains), `'___'` (exactly 3 chars)

#### 📊 Ordering Results
```sql
SELECT DISTINCT name FROM instructor ORDER BY dept_name ASC, name DESC;
```
- `ASC` is default. Sorts left-to-right.

#### 🧮 Set Operations (`UNION`, `INTERSECT`, `EXCEPT`)
- Automatically **remove duplicates**. Use `ALL` variants to keep them.
- **Requirement:** Same number of columns & compatible data types.
  ```sql
  (SELECT course_id FROM section WHERE sem='Fall' AND year=2017)
  UNION
  (SELECT course_id FROM section WHERE sem='Spring' AND year=2018);
  ```

#### ❓ Null Values & Three-Valued Logic
- `NULL` = unknown or inapplicable.
- Arithmetic with `NULL` → `NULL`. Comparison with `NULL` → `UNKNOWN`.
- `WHERE` treats `UNKNOWN` as `FALSE`.
- Test explicitly: `WHERE salary IS NULL` or `IS NOT NULL`
- **Boolean Logic with UNKNOWN:**
  - `TRUE AND UNKNOWN` = `UNKNOWN`
  - `FALSE AND UNKNOWN` = `FALSE`
  - `TRUE OR UNKNOWN` = `TRUE`

### 🔹 5. Aggregate Functions & Grouping
| Function | Meaning |
|----------|---------|
| `AVG()`, `MIN()`, `MAX()`, `SUM()` | Standard math aggregates |
| `COUNT(*)` | Counts all rows |
| `COUNT(DISTINCT col)` | Counts unique non-null values |

#### 📦 `GROUP BY`
- Groups rows by attribute(s) **before** aggregation.
- ⚠️ **Rule:** Any column in `SELECT` that isn't inside an aggregate **must** appear in `GROUP BY`.
  ```sql
  SELECT dept_name, AVG(salary) AS avg_sal 
  FROM instructor GROUP BY dept_name;
  ```

#### 🔍 `HAVING` Clause
- Filters **groups** (applied *after* aggregation), unlike `WHERE` which filters rows *before*.
  ```sql
  SELECT dept_name, AVG(salary) AS avg_sal 
  FROM instructor GROUP BY dept_name 
  HAVING AVG(salary) > 42000;
  ```

### 🔹 6. Nested Subqueries & Advanced Constructs
Subqueries can appear in `FROM`, `WHERE`, or `SELECT`.

#### 🎯 Set Membership & Comparison
| Operator | Meaning | Example |
|----------|---------|---------|
| `IN` / `NOT IN` | Value exists in subquery result | `WHERE course_id IN (SELECT ...)` |
| `> SOME` | True if > at least one value | `WHERE salary > SOME (SELECT salary FROM instructor WHERE dept_name='Biology')` |
| `> ALL` | True if > every value | `WHERE salary > ALL (SELECT salary FROM ...)` |

#### 🔎 Existence & Uniqueness
- `EXISTS (subquery)`: True if subquery returns ≥1 row.
- `NOT EXISTS`: True if subquery returns 0 rows. Often used for **relational division** (e.g., "students who took all Biology courses").
- `UNIQUE (subquery)`: True if subquery result contains no duplicates.

#### 📝 `WITH` Clause (Common Table Expressions)
- Creates temporary named relations for cleaner, readable queries.
  ```sql
  WITH max_budget(value) AS (SELECT max(budget) FROM department)
  SELECT name FROM department, max_budget 
  WHERE department.budget = max_budget.value;
  ```

#### 🔢 Scalar Subqueries
- Expected to return **exactly one value**. Can be used in `SELECT` or `WHERE`.
  ```sql
  SELECT dept_name, 
    (SELECT count(*) FROM instructor WHERE department.dept_name = instructor.dept_name) 
    AS num_instructors FROM department;
  ```
- ⚠️ Runtime error if it returns >1 row.

### 🔹 7. Database Modification (DML)
#### 🗑️ `DELETE`
- Subqueries are evaluated **once** before deletion begins to avoid "moving target" problems.
  ```sql
  DELETE FROM instructor WHERE salary < (SELECT avg(salary) FROM instructor);
  ```

#### ➕ `INSERT`
- Can insert a single row or the entire result of a query.
  ```sql
  INSERT INTO instructor 
  SELECT ID, name, dept_name, 18000 FROM student 
  WHERE dept_name='Music' AND total_cred > 144;
  ```
- Query is fully evaluated before insertion to prevent infinite loops.

#### ✏️ `UPDATE`
- Modifies existing rows. Use `CASE` for conditional, set-based updates.
  ```sql
  UPDATE instructor SET salary = 
    CASE 
      WHEN salary <= 100000 THEN salary * 1.05 
      ELSE salary * 1.03 
    END;
  ```
- Updates with correlated scalar subqueries can recompute derived fields (e.g., `tot_cred` for students).

---

### 💡 Key Takeaways
1. **SQL is declarative & relational-algebra based:** `SELECT` ≈ projection, `FROM` ≈ product/join, `WHERE` ≈ selection.
2. **Duplicates & Nulls are first-class citizens:** SQL operates on multisets, and `NULL` introduces three-valued logic that changes how filters evaluate.
3. **Aggregation splits rows into groups:** `GROUP BY` defines buckets, `HAVING` filters those buckets, and aggregate functions compute summaries.
4. **Subqueries unlock complex logic:** `IN`, `SOME/ALL`, `EXISTS`, and `WITH` let you express queries that would require multiple steps in procedural languages.
5. **Modifications are set-based & safe:** `DELETE` and `UPDATE` evaluate conditions before applying changes, and `CASE` enables powerful conditional updates in a single pass.
6. **Real-world usage:** SQL handles data retrieval & modification, but application logic, UI, and networking belong in host languages connected via JDBC/ODBC.
