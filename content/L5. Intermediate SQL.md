### 🔹 1. Join Expressions
Joins combine rows from two or more tables based on a related column. SQL provides several join flavors to handle different matching requirements.

#### 🔄 Natural Join
- Matches tuples on **all common attributes** and keeps only one copy of each common column.
- *Example:* 
  ```sql
  SELECT name, course_id 
  FROM student NATURAL JOIN takes;
  ```
- ⚠️ **Danger:** If unrelated attributes share the same name, they get incorrectly equated.  
  *Example:* `student NATURAL JOIN takes NATURAL JOIN course` might incorrectly equate `dept_name` (if both `student` and `course` have it), silently dropping valid rows where a student takes a course outside their department.  
  ✅ **Fix:** Use explicit join conditions or limit natural joins to carefully designed schemas.

#### 🌐 Outer Joins (Preserve Non-Matching Tuples)
Inner/natural joins drop rows that don't match. Outer joins keep them, filling missing sides with `NULL`.

|Type|What it keeps|SQL Syntax|
|---|---|---|
|**Left**|All left tuples + matching right|`course NATURAL LEFT OUTER JOIN prereq`|
|**Right**|All right tuples + matching left|`course NATURAL RIGHT OUTER JOIN prereq`|
|**Full**|All tuples from both sides|`course FULL OUTER JOIN prereq USING(course_id)`|
#### 🎯 Explicit Join Conditions
Instead of relying on column names, you can specify exactly how to match:
```sql
-- Inner join with condition
FROM course INNER JOIN prereq ON course.course_id = prereq.course_id

-- Left outer join with condition
FROM course LEFT OUTER JOIN prereq ON course.course_id = prereq.course_id
```

### 🔹 2. Views: Virtual Relations
A **view** is a named query stored in the database. It acts as a virtual table, hiding complexity or sensitive data.

#### 📖 Definition & Expansion
```sql
CREATE VIEW faculty AS 
SELECT ID, name, dept_name FROM instructor;
```
- When you query `faculty`, the DB performs **view expansion**: it substitutes the view definition into your query before execution.
- Views can be built on other views (dependency chains). As long as there's no recursion, expansion terminates cleanly.

#### 💾 Materialized Views
- Some DBMSs physically store the result of a view to speed up repeated queries.
- ⚠️ **Trade-off:** The stored copy becomes stale when base tables change, requiring automatic or manual refresh/maintenance.

#### 🛑 Updating Views
Updates (`INSERT`, `UPDATE`, `DELETE`) on views are **not always possible** because the DB can't always translate them back to base tables uniquely.
- ✅ **Allowed:** Simple views (single table, no aggregates, no `DISTINCT`, missing columns can be `NULL`).
- ❌ **Rejected/Ambiguous:** 
  - Joins (which table gets the new row?)
  - Aggregates/`GROUP BY` (can't reverse a sum)
  - Missing `NOT NULL` columns in the base table
  - `INSERT` into a view filtered by `WHERE dept='History'` that tries to insert a `'Biology'` row (violates the view's own logic)

### 🔹 3. Transactions
A transaction is a **unit of work** containing one or more SQL statements. It guarantees consistency even during crashes or concurrent access.
- **Starts:** Implicitly with the first SQL statement.
- **Ends with:**
  - `COMMIT WORK;` → Changes become permanent.
  - `ROLLBACK WORK;` → All changes in the transaction are undone.
- Ensures **Atomicity** (all-or-nothing) and **Isolation** (concurrent transactions don't interfere).

### 🔹 4. Integrity Constraints
Constraints prevent invalid or inconsistent data from entering the database.

#### 📦 Single-Relation Constraints
| Constraint | Purpose | Example |
|------------|---------|---------|
| `NOT NULL` | Prevents missing values | `name VARCHAR(20) NOT NULL` |
| `UNIQUE` | Ensures no duplicate values (allows `NULL`) | `UNIQUE(email)` |
| `CHECK(P)` | Enforces a condition on every tuple | `CHECK(semester IN ('Fall','Winter','Spring','Summer'))` |

#### 🔗 Referential Integrity & Foreign Keys
Ensures values in one table exist in another.
```sql
FOREIGN KEY(dept_name) REFERENCES department
```
- **Default behavior:** Rejects `INSERT`/`UPDATE`/`DELETE` that would break the link.
- **Cascading Actions:** Automatically propagate changes:
  ```sql
  FOREIGN KEY(dept_name) REFERENCES department
    ON DELETE CASCADE   -- Delete child rows when parent is deleted
    ON UPDATE SET NULL  -- Set FK to NULL when parent PK changes
  ```

#### 🔄 Deferring Constraint Checks
Useful for circular references (e.g., a `person` table where `mother` and `father` are FKs to `person`). You can insert both parent and child, then **defer** FK validation until `COMMIT` to avoid mid-transaction violations.

#### 📜 Assertions & Complex Checks
- `CHECK` with subqueries must be validated on updates to **multiple** tables.
- **Assertions** enforce database-wide conditions:
  ```sql
  CREATE ASSERTION salary_cap 
  CHECK (NOT EXISTS (SELECT * FROM instructor WHERE salary > 200000));
  ```
  *(Note: Rarely implemented in commercial DBs due to performance overhead, but conceptually important.)*

### 🔹 5. SQL Data Types & Schemas
Beyond basic `int`/`varchar`, SQL supports specialized types:

| Type Category | Examples & Usage |
|---------------|------------------|
| **Temporal** | `DATE '2026-05-11'`, `TIME '09:30:00'`, `TIMESTAMP '2026-05-11 09:30:00'`, `INTERVAL '1' DAY` |
| **Large Objects** | `BLOB` (binary: images, videos), `CLOB` (character: large text). Queries return a pointer, not the full object. |
| **User-Defined Types** | `CREATE TYPE Dollars AS NUMERIC(12,2) FINAL;` → `budget Dollars` |
| **Domains** | `CREATE DOMAIN Degree_Level VARCHAR(10) CONSTRAINT valid_deg CHECK (VALUE IN ('Bachelors','Masters','Doctorate'));` |

### 🔹 6. Index Definition in SQL
Indexes speed up lookups by avoiding full table scans.
```sql
CREATE INDEX studentID_index ON student(ID);
```
- **How it works:** Creates a lookup structure (typically B-tree) mapping `ID` values to disk locations.
- **Trade-off:** Faster `SELECT`/`WHERE` queries, but slower `INSERT`/`UPDATE`/`DELETE` (index must be maintained).

### 🔹 7. Authorization & Security
Controls who can do what in the database.

#### 🔑 Privileges & Granting
| Privilege | Allows |
|-----------|--------|
| `SELECT` | Read data or query a view |
| `INSERT` | Add new tuples |
| `UPDATE` | Modify existing tuples |
| `DELETE` | Remove tuples |
| `REFERENCES` | Create foreign keys pointing to the table |
| `ALL PRIVILEGES` | Grants every allowable privilege |

```sql
GRANT SELECT, UPDATE(salary) ON instructor TO amit, satoshi;
REVOKE DELETE ON student FROM u1, u2;
```

#### 👥 Roles
Roles group privileges for easier management:
```sql
CREATE ROLE instructor;
GRANT SELECT ON takes TO instructor;
GRANT instructor TO amit;
-- Roles can inherit from other roles:
CREATE ROLE teaching_assistant;
GRANT teaching_assistant TO instructor;
```

#### 🔐 Advanced Authorization Features
- **View Security:** Granting `SELECT` on a view does **not** grant access to underlying tables. The view creator must have the necessary base-table permissions.
- **`WITH GRANT OPTION`:** Allows a user to pass granted privileges to others.
- **Revoke Behavior:**
  - `REVOKE ... CASCADE` → Revokes the privilege and all dependent grants.
  - `REVOKE ... RESTRICT` → Fails if other users depend on the privilege.

---

### 💡 Key Takeaways
1. **Joins** are powerful but require care: `NATURAL JOIN` is convenient but risky with overlapping column names. Explicit `ON`/`USING` clauses are safer.
2. **Outer joins** prevent data loss when matching is incomplete by padding with `NULL`s.
3. **Views** provide security and abstraction but are often read-only. Updates only work on simple, unambiguous views.
4. **Constraints** (`CHECK`, `FOREIGN KEY`, `ASSERTION`) enforce data correctness at the database level, reducing application-side validation.
5. **Transactions** wrap multiple operations into atomic units, guaranteeing consistency via `COMMIT`/`ROLLBACK`.
6. **Indexes** dramatically speed up queries but add overhead to writes.
7. **Authorization** uses a fine-grained privilege model + roles for scalable security. `GRANT OPTION` and cascade/revoke policies control privilege delegation.
