### 🔹 1. What is Normalization & Why Do We Need It?
Normalization is the systematic process of organizing database tables to **minimize redundancy** and **prevent data anomalies**.

| Anomaly Type | Problem | Example |
|--------------|---------|---------|
| **Update Anomaly** | Changing data in one row but forgetting to update it elsewhere → inconsistency | Updating a professor's office number in one table but not another |
| **Insert Anomaly** | Unable to add valid data because other unrelated data is missing | Can't add a new course until a student enrolls in it |
| **Delete Anomaly** | Accidentally losing valid data when deleting a row | Deleting a student's enrollment also erases the only record of a course |

✅ **A properly normalized database should have:**
- Scalar (atomic) values in every field
- Minimal redundancy
- Minimal use of `NULL`s
- No loss of essential information

> 💡 Normalization is a **step-by-step process**: tables are progressively restructured to meet stricter "Normal Forms" (1NF → 2NF → 3NF → BCNF → etc.)

---

### 🔹 2. Functional Dependencies (The Foundation)
Before checking normal forms, you must understand **Functional Dependencies (FDs)**.

- **Definition:** `X → Y` means attribute set `X` **uniquely determines** attribute set `Y`. For a given value of `X`, there is exactly one corresponding value of `Y`.
- **Example 1:** `AuId → AuAddress` (An author's ID determines their address)
- **Example 2:** `{Title, PubId} → Price` (Book title + publisher ID determines the price)
- **Why it matters:** FDs tell us which attributes naturally belong together. Violating them causes redundancy.

---

### 🔹 3. First Normal Form (1NF)
#### 📏 Rule:
A table is in **1NF** if:
1. Every field contains only **atomic (scalar) values** (no lists, arrays, or repeating groups)
2. Each column has a unique name
3. No duplicate rows exist

#### ⚠️ Problem Example:
| Author | AuPhone |
|--------|---------|
| Smith | 555-0100, 555-0199 |
| Jones | 555-0200 |

`AuPhone` contains multiple values → **Not atomic** → Violates 1NF.

#### 🛠️ Decomposition Steps (Fixing 1NF):
1. Move repeating/multivalued items into a **new table**
2. Assign a **primary key** to the new table
3. Copy the original table's PK into the new table to maintain the relationship

✅ **Result:**
- `Author(AuId, AuName)` → PK: `AuId`
- `Author_Phone(AuId, Phone)` → PK: `(AuId, Phone)`

---

### 🔹 4. Second Normal Form (2NF)
#### 📏 Rules:
1. Table must already be in **1NF**
2. **All non-key attributes must be fully functionally dependent on the ENTIRE primary key** (not just a part of it). *Applies mainly to tables with composite PKs.*

#### ⚠️ Problem Example:
Schema: `{Title, PubId, AuId, Price, AuAddress}`  
PK: `{Title, PubId, AuId}`  
- `Price` depends on the **full PK** ✅  
- `AuAddress` depends **only on `AuId`** (a subset of the PK) ❌ → **Partial Dependency**

#### 🛠️ Decomposition Steps (Fixing 2NF):
1. Identify attributes that depend on only **part** of the composite PK
2. Move those attributes + the partial key they depend on to a **new table**
3. Make that partial key the new table's PK

✅ **Result:**
- `Book_Price(Title, PubId, AuId, Price)` → PK: `{Title, PubId, AuId}`
- `Author_Info(AuId, AuAddress)` → PK: `AuId`

#### 📌 Another Example:
Schema: `{City, Street, HouseNumber, HouseColor, CityPopulation}`  
PK: `{City, Street, HouseNumber}`  
- `CityPopulation` depends only on `City` → Split into:
  - `Address(City, Street, HouseNumber, HouseColor)`
  - `City_Pop(City, CityPopulation)`

---

### 🔹 5. Third Normal Form (3NF)
#### 📏 Rules:
A table is in **3NF** if for every non-trivial FD `X → Y`:
1. `X` is a **superkey**, OR
2. Every attribute in `Y` is **part of a candidate key**

> 💡 **Purpose:** Eliminates **transitive dependencies** (non-key attribute depending on another non-key attribute).

#### ✅ Example (In 3NF):
R(A, B, C) with FDs: `AB → C`, `C → B`  
- Key: `AB`
- `AB → C` → OK (`AB` is a key)
- `C → B` → OK (`B` is part of a key)
✅ Table is in 3NF.

#### ❌ Example (NOT in 3NF):
R(A, B, C) with FDs: `A → B`, `B → C`  
- Key: `A`
- `A → B` → OK (`A` is a key)
- `B → C` → ❌ Violates 3NF! `B` is not a superkey, and `C` is not part of any key.  
*(This is a transitive dependency: `A → B → C`)*

#### 🛠️ Fix for 3NF:
Decompose into two tables to break the chain:
- `R1(A, B)` → PK: `A`
- `R2(B, C)` → PK: `B`

---

### 🔹 6. Beyond 3NF (Brief Overview)
Normalization continues with stricter forms, though **3NF and BCNF** cover 95% of real-world design needs:

| Normal Form | Focus                                                          |
| ----------- | -------------------------------------------------------------- |
| **BCNF**    | Stronger than 3NF: _Every_ determinant must be a candidate key |
| **4NF**     | Eliminates multi-valued dependencies                           |
| **5NF**     | Eliminates join dependencies                                   |
| **DKNF**    | Domain-Key Normal Form (theoretical ideal)                     |

---

### 💡 Key Takeaways
1. **Normalization is progressive:** 1NF → 2NF → 3NF → BCNF. Each step removes a specific type of redundancy.
2. **Functional Dependencies (X → Y)** are the mathematical foundation for checking normal forms.
3. **1NF** = Atomic values (no repeating groups).
4. **2NF** = No partial dependencies (non-key attributes must depend on the *full* composite key).
5. **3NF** = No transitive dependencies (non-key attributes must depend *only* on keys).
6. **Decomposition rules** are systematic: identify the violating FD, split the table, preserve keys, and maintain relationships.
7. **Trade-off:** Over-normalization can lead to excessive joins and slower queries. In practice, designers often stop at 3NF and selectively denormalize for performance.
