### 🔹 1. What is Normalization & Why Do We Need It?
Normalization is a systematic database schema design technique that organizes tables to **minimize data redundancy** and **eliminate update/insert/delete anomalies**.
- **Origin:** Developed by IBM researcher E.F. Codd in the 1970s.
- **Core Idea:** Split large, complex tables into smaller, logically linked tables.
- **Goal:** Achieve an optimal structure where data is consistent, dependencies are logical, and storage is efficient.

---

### 🔹 2. The Core Problem: Database Anomalies
Poorly designed tables lead to three critical anomalies. *Example from the slides: An `Employee` table tracking `EmpId`, `EmpName`, `EmpDept`, `EmpAddress`.*

| Anomaly Type | Problem | Concrete Example |
|--------------|---------|------------------|
| **Update Anomaly** | Modifying data requires multiple row updates. Missing one creates inconsistency. | Employee `100` works in 2 departments. Updating their address requires changing 2 rows. If one is missed, the DB shows two different addresses for the same person. |
| **Insert Anomaly** | Unable to add valid data because required related data is missing. | A new employee joins but isn't assigned to a department yet. If `EmpDept` is `NOT NULL`, you cannot insert their record. |
| **Delete Anomaly** | Deleting a row accidentally removes other valid, unrelated data. | Department `103` closes. Deleting all rows with `EmpDept=103` also erases the only record of employee `Peter`, who worked exclusively there. |

✅ **Normalization solves these by ensuring each fact is stored in exactly one place.**

---

### 🔹 3. The Normal Forms (Step-by-Step Progression)
Normalization is applied progressively. Each form builds on the previous one.

#### 📦 First Normal Form (1NF)
| Rule | Explanation |
|------|-------------|
| **Atomic Values** | Every cell must contain a single, indivisible value. No lists, arrays, or repeating groups. |
| **Unique Rows** | No duplicate tuples. |
| **Unique Column Names** | Each attribute has a distinct name. |

🔍 **Example:**

|Employee|Age|Department|
|---|---|---|
|Melvin|32|Marketing, Sales|

✅ **Fix (Decomposition):** Split into multiple rows so each cell is atomic:

|Employee|Age|Department|
|---|---|---|
|Melvin|32|Marketing|
|Melvin|32|Sales|

---

#### 🔗 Second Normal Form (2NF)
| Rule | Explanation |
|------|-------------|
| **Must be in 1NF** | Prerequisite. |
| **No Partial Dependency** | Every non-prime attribute must depend on the **entire** candidate key, not just a part of it. |

📌 **Key Terms:**
- **Prime Attribute:** Part of any candidate key (e.g., `Emp_Id`, `Dept_Id`).
- **Non-Prime Attribute:** Not part of any key (e.g., `Emp_Name`, `Dept_Name`).

🔍 **Example:** Table with composite key `(Emp_Id, Dept_Id)`
- `Emp_Name` depends only on `Emp_Id` → ❌ Partial Dependency
- `Dept_Name` depends only on `Dept_Id` → ❌ Partial Dependency

✅ **Fix (Decomposition):**
- `Employee(Emp_Id, Emp_Name)`
- `Department(Dept_Id, Dept_Name)`
- `Emp_Dept_Assignment(Emp_Id, Dept_Id)`

---

#### 🔄 Third Normal Form (3NF)
| Rule | Explanation |
|------|-------------|
| **Must be in 2NF** | Prerequisite. |
| **No Transitive Dependency** | Non-prime attributes must not depend on other non-prime attributes. They should depend **only** on the primary key. |

🔍 **Example:** `Employee(Emp_Id, Emp_Name, ZIP, City)`
- `Emp_Id → ZIP` (Valid)
- `ZIP → City` (Valid)
- Therefore, `Emp_Id → City` **transitively** via `ZIP` → ❌ Transitive Dependency

✅ **Fix (Decomposition):**
- `Employee(Emp_Id, Emp_Name, ZIP)`
- `Location(ZIP, City)`

---

#### 🛡️ Boyce-Codd Normal Form (BCNF)
- A stricter version of 3NF designed to handle edge cases where overlapping candidate keys cause anomalies.
- **Rule:** For every functional dependency `X → Y`, `X` must be a **superkey**.
- If a table is in 3NF and has only one candidate key, it is automatically in BCNF.

### 🔹 4. Advantages & Disadvantages of Normalization
| ✅ Advantages | ❌ Disadvantages |
|---------------|------------------|
| **Reduced Size:** Eliminates duplicate data, saving storage. | **More Joins:** Querying spread-out data requires complex `JOIN` operations. |
| **Better Write Performance:** Updates/inserts touch fewer rows and are faster. | **Lookup Overhead:** Real data is often replaced with codes/IDs, requiring extra table lookups. |
| **Fewer Indexes:** Narrower tables mean faster index maintenance. | **Ad-Hoc Query Difficulty:** Optimized for transactional apps, not flexible reporting or analytics. |
| **Data Consistency:** One source of truth prevents update anomalies. | **Design Complexity:** Requires deep understanding of FDs and normal forms; mistakes cause worse designs. |

> 💡 **Trade-off Note:** In practice, databases are often normalized to **3NF** for transactional systems (OLTP), but strategically **denormalized** for data warehouses/analytics (OLAP) to speed up read-heavy queries.

---

### 🔹 5. Key Dependency Concepts Explained
| Concept | Definition | Fixed By |
|---------|------------|----------|
| **Functional Dependency (FD)** | `X → Y` means knowing `X` uniquely determines `Y`. | Foundation of all NFs |
| **Partial Dependency** | Non-key attribute depends on only *part* of a composite key. | 2NF |
| **Transitive Dependency** | Non-key `A` determines non-key `B`, which determines non-key `C` (`A → B → C`). | 3NF |

---

### 💡 Key Takeaways
1. **Normalization is progressive:** 1NF → 2NF → 3NF → BCNF. Each step removes a specific type of redundancy.
2. **Anomalies are the enemy:** Update, Insert, and Delete anomalies occur when data is improperly grouped. Normalization eliminates them.
3. **Dependencies dictate design:** 
   - `1NF` = Atomic cells
   - `2NF` = Full key dependency
   - `3NF` = Direct key dependency only (no transitive chains)
   - `BCNF` = Every determinant must be a superkey
4. **Trade-offs exist:** While normalization guarantees consistency and efficient writes, it increases join complexity for reads. Real-world DBAs balance normalization with performance needs.
