### 🔹 1. Semi-Structured Data
The strict relational model (fixed schema, atomic values) is sometimes too rigid for modern applications. Semi-structured data offers flexibility where schemas evolve frequently or data naturally contains nested/multi-valued elements.

| Feature | Explanation | Example |
|---------|-------------|---------|
| **Flexible Schema** | Tuples can have different attributes; new fields added anytime without schema migrations | User profiles with optional `nickname`, `bio`, or `preferences` |
| **Wide/Sparse Columns** | Table has many possible columns, but each row only stores a subset | IoT sensor logs where not every sensor reports every metric |
| **Multivalued Types** | Sets, multisets, and key-value maps stored directly | `interests = {'basketball', 'anime', 'cooking'}`<br>`device = {(brand, Apple), (size, 13), (color, silver)}` |
| **Arrays** | Ordered sequences, ideal for time-series or scientific readings | `[5, 8, 9, 11]` instead of `(1,5), (2,8), (3,9), (4,11)` |

> 💡 **Why it matters:** Avoids over-normalization. Storing a user's tags as an array is simpler than creating a separate `user_tags` table and joining constantly.

### 🔹 2. Nested Data Formats: JSON & XML
These are the industry standards for representing hierarchical, semi-structured data.

#### 🟨 JSON (JavaScript Object Notation)
- **Structure:** Objects `{}` (key-value maps) + Arrays `[]` + Scalars (strings, numbers, booleans)
- **Ubiquity:** Default format for web APIs, mobile apps, and NoSQL/document databases
- **SQL Support:** Modern RDBMS natively support `JSON` types with:
  - Path extraction: `data->'name'` or `data->'ID'`
  - Aggregation: `json_agg()` to build JSON from relational rows
  - Storage optimization: Binary JSON (BSON) reduces verbosity
- *Example:*
  ```json
  {
    "ID": "22222",
    "name": {"first": "Albert", "last": "Einstein"},
    "children": [
      {"first": "Hans", "last": "Einstein"},
      {"first": "Eduard", "last": "Einstein"}
    ]
  }
  ```

#### 🟦 XML (Extensible Markup Language)
- **Structure:** Tag-based, hierarchical, self-documenting
- **Use Case:** Complex documents, legacy enterprise systems, configuration files
- **Querying:** `XQuery` (specialized) or SQL/XML extensions with path expressions
- *Example:* `<course><course_id>CS-101</course_id><title>Intro to CS</title></course>`

### 🔹 3. Object Orientation & ORM
Object-Oriented (OO) languages (Java, Python, C++) don't naturally map to relational tables. Three integration approaches exist:

| Approach | How it Works | Trade-off |
|----------|--------------|-----------|
| **Object-Relational DB** | Extends SQL with OO features: user-defined types, inheritance, references | Keeps data in DB, but SQL becomes complex |
| **ORM (Object-Relational Mapping)** | Framework auto-maps classes ↔ tables, handles CRUD operations | High developer productivity, but can hide inefficient queries |
| **Native OO Database** | Stores objects directly, accessed from OO language | Rarely used today due to ecosystem lock-in |

#### 🔑 Key Object-Relational SQL Features:
- **User-Defined Types:** `CREATE TYPE Person (ID varchar, name varchar) ref from(ID);`
- **Table Inheritance:** `CREATE TABLE students UNDER people;` (Students inherit all Person attributes)
- **Reference Types & Path Expressions:** `head REF(Person) scope people;` → Query as `SELECT head->name FROM departments;`
- **Popular ORMs:** Hibernate (Java), Django ORM / SQLAlchemy (Python)

### 🔹 4. Textual Data & Information Retrieval (IR)
Unlike exact relational queries, IR handles **unstructured text** and ranks results by relevance.

| Concept | Explanation | Formula / Metric |
|---------|-------------|------------------|
| **Keyword Query** | Returns documents containing all query terms | `"cricket"` → returns match info, articles, stats |
| **TF-IDF Ranking** | Measures relevance of term `t` to document `d` | `TF(d,t) = log(1 + n(d,t)/n(d))`<br>`IDF(t) = 1/n(t)`<br>`Relevance = Σ TF × IDF` |
| **Precision** | % of returned results that are actually relevant | High precision = fewer false positives |
| **Recall** | % of all relevant results that were returned | High recall = fewer false negatives |
| **Structured Keyword Search** | Matches keywords across tables, returns connected subgraphs | Query `"Zhang Katz"` → finds student `Zhang`, instructor `Katz`, and their `advisor` link |

> 💡 **Why TF-IDF works:** Common words ("the", "and") get low IDF scores, so they don't dominate relevance. Rare, meaningful terms get higher weights.

### 🔹 5. Spatial Data
Spatial databases store, index, and query geographic or geometric information efficiently.

| Data Type | Description | Representation |
|-----------|-------------|----------------|
| **Geographic** | Earth-surface data (roads, boundaries, elevation) | `(Latitude, Longitude, Elevation)` |
| **Geometric** | CAD/design data (buildings, circuits, aircraft) | 2D/3D Euclidean `(X, Y, Z)` coordinates |
| **Raster** | Pixel/bitmaps (satellite images, temperature grids) | Grid of values per pixel |
| **Vector** | Mathematical shapes (points, lines, polygons) | `LINESTRING(1 1, 2 3, 4 4)`<br>`POLYGON((1 1, 2 3, 4 4, 1 1))` |

#### 🌍 Spatial Query Types:
| Query | Purpose | Example (PostGIS/SQL Server) |
|-------|---------|------------------------------|
| **Region** | Find objects inside/overlapping a shape | `ST_Contains(polygon, point)` |
| **Nearness** | Objects within distance `X` of a location | `ST_DWithin(loc1, loc2, 500)` |
| **Nearest Neighbor** | Closest object meeting a condition | "Find nearest hospital to accident site" |
| **Spatial Join** | Join two tables based on location overlap | `SELECT * FROM roads, parcels WHERE ST_Intersects(roads.geom, parcels.geom)` |
| **Graph/Network** | Shortest path via road/transit network | GPS routing algorithms |

> 💡 **Operations:** `ST_Union()`, `ST_Intersection()`, `ST_GeometryFromText()` convert between human-readable text and optimized binary spatial formats.

---

### 💡 Key Takeaways
1. **Semi-structured data** bridges the gap between rigid relational schemas and real-world flexible data (profiles, logs, IoT).
2. **JSON & XML** are the dominant nested formats; modern SQL engines natively parse, query, and aggregate them.
3. **Object-Relational features** (types, inheritance, references) and **ORMs** solve the "impedance mismatch" between code objects and database tables.
4. **Textual IR** relies on relevance ranking (TF-IDF) rather than exact matches, measured by precision and recall.
5. **Spatial databases** handle vector/raster data with specialized indexes and operators (`ST_*`) for geographic, CAD, and routing applications.
6. **SQL has evolved** beyond simple tables to become a unified query engine for relational, JSON, text, and spatial data.

> 🎯 **Next Step:** In production systems, you'll choose the right data model per use case: relational for transactions, JSON for app APIs, full-text indexes for search, and spatial extensions (PostGIS, SQL Server Spatial) for location services. Understanding these complex types prepares you for modern multi-model database architecture and NoSQL integration.
