### 🔹 1. The Query Processing Pipeline
Every SQL query goes through three standardized stages before returning results:

| Stage                     | What Happens                                                                                            | Example                                                                                   |
| ------------------------- | ------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| **Parsing & Translation** | Checks syntax, verifies table/column names, and converts SQL to internal relational algebra             | `SELECT name FROM instructor WHERE salary < 75000` → `π_name(σ_salary<75000(instructor))` |
| **Optimization**          | Generates multiple logically equivalent evaluation plans and estimates their costs to pick the cheapest | Choosing to use a salary index instead of scanning the whole `instructor` table           |
| **Evaluation**            | The execution engine runs the chosen plan step-by-step and returns the result tuples                    | Fetches records, applies filters, joins, and projects columns                             |
> 💡 **Key Insight:** A poorly optimized plan can take *days* instead of *seconds*. The optimizer's job is to find the fastest path without changing the logical result.

---

### 🔹 2. Measuring Query Cost
Query cost is dominated by **disk I/O**. CPU and network costs are ignored for simplicity in theory (though real systems account for them).
- **Formula:** `Cost = (b × tT) + (S × tS)`
  - `b` = number of block transfers
  - `tT` = time to transfer one block (~0.1 ms HDD, ~5 μs SSD)
  - `S` = number of disk seeks
  - `tS` = time per seek (~4 ms HDD, ~30 μs SSD)
- **Assumption:** Worst-case scenario where no data is initially in the buffer manager.

### 🔹 3. Selection Algorithms (`σ`)
How the DB finds rows matching a `WHERE` clause:

| Algorithm | When to Use | Cost Estimate | Trade-off |
|-----------|-------------|---------------|-----------|
| **A1: Linear Scan** | No index, or condition on non-indexed column | `br` transfers (+1 seek) | Simple, works for any condition |
| **A2: Clustering Index (Key)** | Equality on a key with clustered index | `(hi+1) × (tT+tS)` | Extremely fast for exact matches |
| **A3: Clustering Index (Non-Key)** | Equality on non-key, records stored together | `hi×(tT+tS) + tS + tT×b` | Efficient because matching blocks are contiguous |
| **A4: Secondary Index** | Equality on non-clustered attribute | `(hi+n) × (tT+tS)` | Expensive if `n` (matching rows) is large → random I/O per row |
| **A5/A6: Comparison (`≥`, `≤`)** | Range queries with clustering index | Scan from first match onward | Avoids full table scan, but still reads sequentially |

> 🎯 **Rule of Thumb:** If a selection is highly selective (returns few rows), an index wins. If it's unselective (returns >10-20% of rows), a linear scan is often faster due to sequential read efficiency.

---

### 🔹 4. External Sorting (When Data Exceeds RAM)
Databases use **Sort-Merge** to sort datasets larger than main memory (`M` pages):
1. **Run Generation:** Read `M` blocks, sort in memory, write to disk as a sorted "run". Repeat until all data is processed → `N` runs.
2. **Merge Passes:** 
   - If `N < M`: Merge all runs in one pass.
   - If `N ≥ M`: Merge `(M-1)` runs at a time. Each pass reduces runs by a factor of `(M-1)` and increases run size.
- *Example:* `M=11`, `90` initial runs → Pass 1 yields `9` larger runs → Pass 2 yields `1` fully sorted file.

---

### 🔹 5. Join Algorithms (The Core of Query Execution)
Joins are the most expensive operations. The optimizer picks one based on data size, memory, and available indexes.

| Algorithm | How It Works | Best Use Case | Cost Characteristic |
|-----------|--------------|---------------|---------------------|
| **Nested-Loop** | `for each r in outer: for each s in inner: check condition` | Small relations, no indexes | `nr × bs + br` (catastrophic if both large) |
| **Block Nested-Loop** | Processes block-by-block instead of tuple-by-tuple | General purpose, fits in memory | Drastically reduces I/O if inner relation fits in buffer |
| **Indexed Nested-Loop** | Uses index on inner relation's join attribute | Inner relation indexed, outer small | `br×(tT+tS) + nr×c` (very fast if index exists) |
| **Merge-Join** | Sort both tables on join key → merge like merge-sort | Equi-joins, data already sorted | `br + bs` + sorting cost. Highly efficient post-sort |
| **Hash-Join** | Hash both relations into partitions → build hash table on smaller, probe with larger | Equi-joins, large unsorted data | Linear I/O. Requires memory for partitions. Handles overflow via recursive partitioning or fallback to block NL |

> ⚠️ **Hash-Join Overflow:** If a partition doesn't fit in memory, the DB recursively re-partitions it. If duplicates cause massive skew, it falls back to Block Nested-Loop for that partition.

### 🔹 6. Expression Evaluation Strategies
How intermediate results are handled in a query tree:

| Strategy                         | Mechanism                                                                      | Pros                                                       | Cons                                                                    |
| -------------------------------- | ------------------------------------------------------------------------------ | ---------------------------------------------------------- | ----------------------------------------------------------------------- |
| **Materialization**              | Execute operator → write result to temp table on disk → next operator reads it | Simple, works for all queries, allows complex optimization | High I/O cost, temp table overhead                                      |
| **Pipelining**                   | Pass tuples directly between operators without writing to disk                 | Near-zero I/O for intermediates, faster execution          | Not always possible (e.g., requires full input for sorting/aggregation) |
| **Demand-Driven (Lazy/Pull)**    | Parent operator requests next tuple → child computes on demand                 | Memory efficient, easy to implement with iterators         | Requires state management between calls                                 |
| **Producer-Driven (Eager/Push)** | Child pushes tuples into buffer → parent consumes when ready                   | Better CPU utilization, overlaps computation               | Requires buffer management & backpressure handling                      |
📌 **Blocking Operations:** Operators like `SORT` or `HASH-JOIN` cannot output anything until all input is read. Optimizers split them into phases (e.g., run generation + merge) to enable partial pipelining.

---

### 🔹 7. Query Optimization & Equivalence Rules
The optimizer rewrites queries into logically equivalent forms to reduce work before execution.

| Rule | Meaning | Why It Matters |
|------|---------|----------------|
| **Decompose Selections** | `σ_θ1∧θ2(E) ≡ σ_θ1(σ_θ2(E))` | Enables pushing filters down individually |
| **Push Selections Down** | `σ_θ(E1 ⋈ E2) ≡ E1 ⋈_θ E2` or push into one side | **Most critical rule.** Filters rows early → smaller joins → less I/O |
| **Projection Merging** | Keep only the final `π` in a chain | Removes unnecessary columns early |
| **Join Commutativity/Associativity** | `E1 ⋈ E2 ≡ E2 ⋈ E1` and `(E1⋈E2)⋈E3 ≡ E1⋈(E2⋈E3)` | Allows reordering joins to minimize intermediate size |

🔍 **Example from Slides:**
```sql
Πname, title(σdept_name="Music"(instructor) ⨝ teaches ⨝ Πcourse_id,title(course))
```
- ❌ Bad order: Join `teaches ⨝ course` first → huge intermediate table.
- ✅ Optimized order: Apply `σdept_name="Music"` to `instructor` first → tiny table → join with `teaches` → join with `course`.

### 🔹 8. Cost Estimation & Dynamic Programming
#### 📊 Catalog Statistics Used:
- Number of tuples (`nr`), number of blocks (`br`)
- Number of distinct values per attribute (`V(A, r)`)
- Used to estimate selectivity and intermediate result sizes.

#### 🌲 Dynamic Programming for Join Ordering:
- Finds the optimal join tree by recursively evaluating all subsets of relations.
- `findbestplan(S)` tries every split `S1 ⋈ (S - S1)` and caches the cheapest plan.
- **Left-Deep Join Trees:** Optimizers often restrict to left-deep trees (right input is always a base relation) because they enable efficient pipelined nested-loop joins and reduce the search space from exponential to manageable.

---

### 🔹 9. Advanced Optimization Features
| Feature | Purpose | How It Works |
|---------|---------|--------------|
| **Parametric Optimization** | Handle runtime parameters (`WHERE price < $1`) | Pre-computes optimal plans for different parameter ranges; selects best at runtime |
| **Plan Stability** | Prevent performance regressions after optimizer upgrades | Allows hints, version-compatible optimization levels, or plan caching from previous versions |
| **Adaptive Query Processing** | Recover from bad cost estimates mid-execution | Monitors actual row counts; switches algorithms on-the-fly (e.g., hash-join → nested-loop) if estimates were wildly wrong |

---

### 💡 Key Takeaways
1. **Query processing is a pipeline:** Parse → Optimize → Execute. The optimizer is where performance lives or dies.
2. **Disk I/O is the bottleneck:** Cost models focus on block transfers and seeks. Sequential access is ~100x faster than random access on HDDs.
3. **Selection pushdown is king:** Filtering rows early dramatically reduces the size of subsequent joins and sorts.
4. **Join choice depends on context:** Hash-join dominates for large equi-joins, merge-join excels when data is pre-sorted, and indexed nested-loop wins when good indexes exist.
5. **Pipelining beats materialization:** Passing tuples directly between operators avoids expensive temp-table writes, but blocking operations (sort, aggregate) limit it.
6. **Dynamic programming finds optimal join orders:** Optimizers use catalog statistics and equivalence rules to explore `2^n` join permutations efficiently, often constraining to left-deep trees for practicality.

> 🎯 **Next Step:** In real-world database engineering, understanding these algorithms helps you write index-aware SQL, interpret `EXPLAIN ANALYZE` output, and diagnose why a query that "should be fast" is actually slow due to poor join ordering, missing statistics, or blocking operations spilling to disk.
