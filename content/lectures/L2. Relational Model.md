### 🔹 1. Structure of Relational Databases
A **relation** is simply a **table** with:
- **Attributes (columns)**: Named fields with a defined domain (set of allowed values).
- **Tuples (rows)**: Individual records; order does **not** matter.

#### 📌 Key Properties:
| Property | Explanation | Example |
|----------|-------------|---------|
| **Atomic values** | Each cell holds a single, indivisible value | `salary = 75000` ✅; `salary = [75000, 80000]` ❌ |
| **Null values** | Special marker meaning "unknown" or "not applicable" | An instructor's `dept_name` might be `null` if unassigned |
| **Unordered tuples** | Rows can be stored/retrieved in any order | Query results don't depend on row position |

*Example `instructor` relation:*

| ID    | name       | dept_name  | salary |
| ----- | ---------- | ---------- | ------ |
| 10101 | Srinivasan | Comp. Sci. | 65000  |
| 12121 | Wu         | Finance    | 90000  |
| 15151 | Mozart     | Music      | 40000  |

### 🔹 2. Database Schema vs. Instance
| Concept | Definition | Analogy | Example |
|---------|------------|---------|---------|
| **Schema** | Logical structure/blueprint of the database | Variable *type* in programming | `instructor(ID, name, dept_name, salary)` |
| **Instance** | Actual data stored at a specific moment | Variable *value* | The table rows shown above |

> 💡 Schema changes rarely; instances change constantly as data is added/updated.

---

### 🔹 3. Keys: Identifying Tuples Uniquely
#### 🗝️ Key Types:
1. **Superkey**: A set of attributes that uniquely identifies a tuple.
   - *Example:* `{ID}` and `{ID, name}` are both superkeys for `instructor`.
2. **Candidate key**: A *minimal* superkey (no unnecessary attributes).
   - *Example:* `{ID}` is a candidate key; `{ID, name}` is not minimal.
3. **Primary key**: The candidate key chosen by the designer to identify tuples.
   - *Convention:* Underlined in schema diagrams → `instructor(**ID**, name, dept_name, salary)`
4. **Foreign key**: An attribute that references the primary key of *another* relation.
   - *Example:* `instructor.dept_name` → references `department.dept_name`
   - Ensures **referential integrity**: You can't assign an instructor to a non-existent department.

*Schema Diagram Snippet:*
```
instructor(**ID**, name, dept_name, salary)
                     │
                     ▼ (foreign key)
department(**dept_name**, building, budget)
```

### 🔹 4. Relational Query Languages
Two main paradigms:

|Type|Description|Example Languages|
|---|---|---|
|**Procedural**|Specify _what_ data + _how_ to get it|Relational Algebra|
|**Declarative**|Specify only _what_ data is needed|SQL, Tuple/Domain Relational Calculus|
> ✅ The "pure" languages (Relational Algebra, Tuple Calculus, Domain Calculus) are **equivalent in power**—anything expressible in one can be expressed in the others.

We focus on **Relational Algebra** because it forms the theoretical foundation for query optimization.

---

### 🔹 5. Relational Algebra: The Six Basic Operators
Each operator takes one or two relations as input and returns a new relation.

#### 🎯 1. **Select (σ)** – Filter rows
- Notation: `σₚ(r)` where `p` is a predicate
- *Example:* Find Physics instructors  
  `σ_dept_name="Physics"(instructor)`
- Supports comparisons: `=, ≠, <, ≤, >, ≥` and logical connectives: `∧ (AND), ∨ (OR), ¬ (NOT)`
- *Complex example:* Physics instructors earning > $90K  
  `σ_dept_name="Physics" ∧ salary>90000(instructor)`

#### 🔍 2. **Project (π)** – Select columns
- Notation: `π_A₁,A₂,…,Aₖ(r)`
- Removes unlisted attributes; **duplicates are eliminated** (relations are sets)
- *Example:* Get only ID, name, and salary  
  `π_ID,name,salary(instructor)`

#### 🔗 3. **Cartesian Product (×)** – Combine all pairs
- Notation: `r × s`
- Pairs every tuple in `r` with every tuple in `s`
- *Caution:* Can create huge results; usually followed by a `select` to filter meaningful combinations.
- *Attribute disambiguation:* If both relations have `ID`, use `instructor.ID` and `teaches.ID`

#### ⚡ 4. **Join (⋈)** – Product + Filter in One Step
- Notation: `r ⋈_θ s` where `θ` is a condition
- Defined as: `σ_θ(r × s)`
- *Most common:* **Equi-join** (condition uses `=`)  
  `instructor ⋈_instructor.ID=teaches.ID teaches`
- *Natural join (⋈)*: Automatically joins on attributes with the same name and removes duplicates.

#### ➕ 5. **Union (∪)** – Combine matching relations
- Notation: `r ∪ s`
- **Requirements**: Same number of attributes + compatible domains
- *Example:* Courses taught in Fall 2017 **or** Spring 2018  
  `π_course_id(σ_semester="Fall"∧year=2017(section)) ∪ π_course_id(σ_semester="Spring"∧year=2018(section))`

#### ➖ 6. **Set Difference (−)** – Find what's in one but not the other
- Notation: `r − s` (same compatibility rules as union)
- *Example:* Courses taught in Fall 2017 **but not** Spring 2018  
  `π_course_id(σ_semester="Fall"∧year=2017(section)) − π_course_id(σ_semester="Spring"∧year=2018(section))`

> 🔄 **Set Intersection (∩)** can be derived: `r ∩ s = r − (r − s)`

### 🔹 6. Additional Useful Operations
#### ✍️ Assignment (←)
- Store intermediate results in temporary variables for clarity.
- *Example:* Instructors in Physics **or** Music:
  ```
  Physics ← σ_dept_name="Physics"(instructor)
  Music   ← σ_dept_name="Music"(instructor)
  Result  ← Physics ∪ Music
  ```

#### 🏷️ Rename (ρ)
- Give a name (and optionally new attribute names) to a relation expression.
- Notation: `ρₓ(E)` or `ρₓ(ₐ₁,ₐ₂,…,ₐₙ)(E)`
- *Use case:* Self-joins or simplifying complex expressions.

#### 📊 Aggregate Functions
Operate on *multisets* (allow duplicates) and return a single value:

|Function|Meaning|Example|
|---|---|---|
|`avg`|Average|`γ_avg(salary)(instructor)` → overall avg salary|
|`sum`|Total|`γ_sum(salary)(instructor)`|
|`count`|Number of values|`γ_count(course_id)(σ_year=2018(section))`|
|`min`/`max`|Extreme values|`γ_min(salary)(instructor)`|
##### 🔁 Grouping with Aggregation
- Notation: `G₁,G₂,…,Gₙ γ_F₁(A₁),…,Fₖ(Aₖ)(r)`
- *Example:* Average salary **per department**:  
  `dept_name γ_avg(salary)(instructor)`

| dept_name | avg_salary |
|-----------|------------|
| Comp. Sci. | 72000 |
| Finance | 85000 |
| Music | 45000 |

### 🔹 7. Query Equivalence & Optimization
- Multiple relational algebra expressions can produce the **same result**.
- *Example:* Find Physics instructors with salary > $90K:
  ```sql
  -- Query 1: Single selection
  σ_dept_name="Physics" ∧ salary>90000(instructor)
  
  -- Query 2: Chained selections (equivalent)
  σ_dept_name="Physics"(σ_salary>90000(instructor))
  ```
- **Why it matters**: The query optimizer chooses the most efficient equivalent form (e.g., applying selective filters early to reduce data size).

- *Join optimization example*:  
  Filtering `instructor` by department *before* joining with `teaches` is usually faster than joining first, then filtering.

---

### 🔹 8. Putting It All Together: A Complex Query
**Goal**: *Find names of instructors in Physics who teach at least one course.*

**Step-by-step in Relational Algebra**:
1. Filter Physics instructors:  
   `σ_dept_name="Physics"(instructor)`
2. Join with `teaches` to link instructors to courses:  
   `(σ_dept_name="Physics"(instructor)) ⋈_instructor.ID=teaches.ID teaches`
3. Project only the names:  
   `π_name( (σ_dept_name="Physics"(instructor)) ⋈_instructor.ID=teaches.ID teaches )`

✅ This demonstrates composition: operations nest because each returns a relation.

---

### 💡 Key Takeaways
1. The **relational model** uses simple, mathematically grounded tables (relations) to represent data.
2. **Keys** enforce uniqueness and relationships; **foreign keys** maintain cross-table consistency.
3. **Relational algebra** provides a formal, procedural foundation for querying—six core operators compose to express complex questions.
4. **Aggregate functions + grouping** enable summarization (e.g., averages per department).
5. **Query equivalence** allows optimizers to rewrite queries for performance without changing results.
6. While users typically write **SQL** (declarative), understanding relational algebra helps write efficient queries and grasp how databases work under the hood.

> 🎯 **Next Step**: In practice, you'll write SQL, but every SQL query is translated into relational algebra internally. Mastering these concepts makes you a better database designer and developer.
