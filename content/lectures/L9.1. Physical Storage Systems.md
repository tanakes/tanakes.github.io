### 🔹 1. Storage Hierarchy: Why It Matters
Databases use multiple storage layers to balance **speed**, **cost**, and **persistence**.

| Storage Type            | Access Time         | Volatility   | Example                |
| ----------------------- | ------------------- | ------------ | ---------------------- |
| **Cache**               | 1-5 nanoseconds     | Volatile     | CPU cache              |
| **Main Memory (RAM)**   | 60-100 nanoseconds  | Volatile     | DDR4/DDR5 RAM          |
| **Flash/SSD**           | 20-100 microseconds | Non-volatile | NVMe SSD, SATA SSD     |
| **Magnetic Disk (HDD)** | 5-10 milliseconds   | Non-volatile | Traditional hard drive |
| **Optical/Tape**        | 100+ milliseconds   | Non-volatile | Blu-ray, LTO tape      |

> 💡 **Key Insight:** Each level is ~10-100x slower but ~10x cheaper per GB than the level above. DBMSs use buffering and caching to keep frequently accessed data in faster storage.

---

### 🔹 2. Magnetic Hard Disk Mechanics
Understanding HDDs helps explain why query performance varies.

#### 📦 Disk Structure:
- **Platters:** Rotating magnetic disks (multiple stacked)
- **Tracks:** Concentric circles on each platter surface
- **Sectors:** Smallest addressable unit (typically 512 bytes or 4KB)
- **Blocks:** Logical unit for DB storage allocation (4-16 KB, multiple sectors)

#### ⏱️ Performance Metrics:
| Metric                 | Definition                           | Typical Value            |
| ---------------------- | ------------------------------------ | ------------------------ |
| **Seek Time**          | Time to move arm to correct track    | 4-10 ms                  |
| **Rotational Latency** | Time for sector to rotate under head | 4-11 ms (5400-15000 RPM) |
| **Transfer Rate**      | Rate of reading/writing data         | 25-200 MB/s              |
| **IOPS**               | Random I/O operations per second     | 50-200 IOPS              |

> 🎯 **Sequential vs. Random Access:** Sequential reads are ~100x faster than random reads because they avoid repeated seeks. DBMSs try to organize data to maximize sequential access.

---

### 🔹 3. Flash Storage (SSDs)
Solid-state drives use NAND flash memory, offering dramatically different performance characteristics.

#### ⚡ Flash Properties:
| Property | Impact on Database Design |
|----------|---------------------------|
| **Page-at-a-time reads** (512B-4KB) | No seek time; random ≈ sequential reads |
| **Write-once per page** | Must erase before rewrite (2-5 ms per erase block) |
| **Erase blocks** (256KB-1MB) | Writes must be batched; wear-leveling required |
| **Limited erase cycles** (1K-100K) | Hot data should be spread across blocks |

#### 📊 SSD Performance vs. HDD:
| Operation | HDD | SSD (SATA) | SSD (NVMe) |
|-----------|-----|------------|------------|
| **Random 4KB Read** | ~100 IOPS | ~10,000 IOPS | ~350,000 IOPS |
| **Random 4KB Write** | ~100 IOPS | ~40,000 IOPS | ~100,000+ IOPS |
| **Sequential Transfer** | 100-200 MB/s | ~500 MB/s | 2-3 GB/s |

> 💡 **Hybrid Disks:** Some systems combine a small flash cache with a larger HDD to get "best of both worlds" for mixed workloads.

---

### 🔹 4. RAID: Redundant Arrays of Independent Disks
RAID combines multiple disks to improve **performance**, **reliability**, or both.

#### 🛡️ Reliability via Redundancy:
| RAID Level | Strategy | Pros | Cons | Best For |
|------------|----------|------|------|----------|
| **RAID 0** | Striping (no redundancy) | Max performance, full capacity | No fault tolerance | Temporary data, caches |
| **RAID 1** | Mirroring (duplicate disks) | Fast writes, simple recovery | 50% storage overhead | Logs, critical small tables |
| **RAID 5** | Block-interleaved distributed parity | Good read performance, efficient storage | Slow random writes (2 reads + 2 writes per update) | General-purpose OLTP |
| **RAID 6** | Dual parity (P+Q) | Tolerates 2 disk failures | Higher write overhead than RAID 5 | Large archives, high-availability systems |

#### 🔢 Reliability Math:
- Single disk MTTF: 1,200,000 hours (~137 years)
- 100-disk array MTTF: ~12,000 hours (~1.4 years) without redundancy
- RAID 1 pair MTTF for *data loss*: ~57,000 years (ignoring correlated failures)

> ⚠️ **Latent Sector Failures:** A disk can silently corrupt data. RAID 5 + one failed disk + one latent error = data loss. RAID 6 or regular scrubbing mitigates this.

### 🔹 5. Buffer Management: Bridging Disk and Memory
The **buffer manager** decides which disk blocks stay in RAM and which get evicted.

#### 🔄 Replacement Policies:
| Policy                        | Strategy                                          | When It Works Best                     |
| ----------------------------- | ------------------------------------------------- | -------------------------------------- |
| **LRU** (Least Recently Used) | Evict block not accessed recently                 | Random access patterns                 |
| **MRU** (Most Recently Used)  | Evict block just finished processing              | Sequential scans (avoids thrashing)    |
| **Toss-Immediate**            | Free block immediately after last tuple processed | One-pass algorithms (e.g., hash joins) |
| **Hint-Based**                | Query optimizer suggests access patterns          | Complex joins with known access order  |

#### ⚠️ LRU Pitfall Example:
In a nested-loop join `R ⋈ S`:
- LRU may evict `S`'s blocks after each `R` tuple is processed
- Result: `S` is re-read from disk for every `R` tuple → catastrophic I/O
- Fix: Pin `S` in memory or use MRU for the inner relation

#### 🔒 Pinning & Locking:
- **Pinned block:** Cannot be evicted (`pin_count > 0`)
- **Shared lock:** Multiple readers allowed
- **Exclusive lock:** Required for writers; blocks other access

### 🔹 6. Disk Access Optimization Techniques
| Technique | Purpose | Example |
|-----------|---------|---------|
| **Buffering** | Cache frequently accessed blocks in RAM | Keep data dictionary pages pinned |
| **Read-Ahead** | Pre-fetch blocks likely to be needed next | Sequential scan: read next 8 blocks proactively |
| **Disk-Arm Scheduling** | Reorder requests to minimize head movement | Elevator algorithm: serve requests in one direction, then reverse |
| **Non-Volatile Write Buffer** | Acknowledge writes instantly, reorder for efficiency | NV-RAM logs writes; flushes to disk in optimal order |
| **Log Disk** | Dedicated disk for sequential write-ahead logging | WAL writes are extremely fast (no seeks) |

### 🔹 7. Storage Interfaces & Modern Hardware
| Interface | Max Throughput | Latency | Use Case |
|-----------|---------------|---------|----------|
| **SATA 3** | 6 Gb/s (~600 MB/s) | ~100 µs | Budget SSDs, HDDs |
| **SAS 3** | 12 Gb/s (~1.2 GB/s) | ~50 µs | Enterprise HDDs/SSDs |
| **NVMe over PCIe** | 24+ Gb/s (~3 GB/s) | ~10 µs | High-performance SSDs |
| **Storage Class Memory (Optane)** | ~6 GB/s | ~1 µs | Persistent memory tier |

> 💡 **SAN vs. NAS:** 
> - **SAN (Storage Area Network):** Block-level access over high-speed network; appears as local disk
> - **NAS (Network Attached Storage):** File-level access over standard network; easier to share but higher latency

---

### 💡 Key Takeaways
1. **Storage hierarchy is fundamental:** DBMSs must manage data across volatile/non-volatile, fast/slow layers to balance cost and performance.
2. **Disk I/O is the bottleneck:** Seek time and rotational latency make random access expensive; sequential access and buffering are critical.
3. **SSDs change the rules:** Near-zero seek time makes random reads cheap, but write amplification and wear-leveling require careful management.
4. **RAID is about trade-offs:** Choose redundancy level based on workload (OLTP vs. archival) and failure tolerance requirements.
5. **Buffer management is query-aware:** LRU isn't always best; modern DBMSs use optimizer hints and workload-aware policies.
6. **Hardware evolves fast:** NVMe, Storage Class Memory, and intelligent controllers continue to reshape storage architecture.

> 🎯 **Next Step:** Understanding physical storage is essential for **index design (Chapter 14)** and **query optimization**. Knowing how data is laid out on disk helps you choose the right indexing strategy, partitioning scheme, and buffer policy for your workload.
